## 1. Назначение документа
Данный артефакт содержит детализированные сценарии взаимодействия пользователя с системой. Спецификации включают предусловия, постусловия, основные и альтернативные потоки, бизнес-правила и маппинг на слои архитектуры PCMEF. Документ используется разработчиками для реализации логики Control/Mediator и тестировщиками для написания сценариев приемочных тестов.

## UC2: Загрузка фотографий в портфолио

| Параметр | Описание |
|----------|----------|
| **ID** | UC2 |
| **Название** | Загрузка фотографий в портфолио |
| **Актор** | Фотограф (авторизованный пользователь) |
| **Краткое описание** | Пользователь выбирает изображения из галереи или снимает на камеру, добавляет их в выбранный альбом. Система валидирует формат/размер, генерирует превью, сохраняет локально (Room) и отправляет на сервер (Retrofit). |

### Предусловия
1. Пользователь успешно авторизован, валидный JWT-токен сохранён в защищённом хранилище
2. В системе существует минимум одно портфолио и один альбом
3. Приложению выданы разрешения `READ_MEDIA_IMAGES` и `CAMERA`

### Постусловия
- **Успех:** Фотографии добавлены в альбом, метаданные сохранены, превью сгенерированы, статус синхронизации установлен в `SYNCED`
- **Неуспех:** Состояние системы не изменено, пользователь получает уведомление об ошибке с рекомендацией

### Триггер
Пользователь нажимает кнопку «Добавить фото» на экране `AlbumDetailScreen`

### Основной поток
1. Система отображает диалог выбора источника: `Галерея` / `Камера`
2. Пользователь выбирает источник и отмечает 1–N изображений
3. Слой **Control** валидирует MIME-тип (`image/jpeg`, `image/png`, `image/webp`) и размер файла (≤ 20 МБ)
4. Слой **Mediator** инициирует генерацию миниатюр (512×512 px) через `ImageDecoder`
5. Система создаёт доменные объекты `Photo` с временными UUID и статусом `PENDING_SYNC`
6. Слой **Foundation** сохраняет записи в `RoomDatabase` через `PhotoDao.insertPending()`
7. `ViewModel` обновляет UI через `StateFlow`, показывая индикатор загрузки
8. Фоновый воркер (`WorkManager`) формирует `MultipartBody` и отправляет пакет на сервер через `Retrofit`
9. Сервер принимает файлы, сохраняет в облачном хранилище, возвращает постоянные ID, URL оригиналов и превью
10. Слой **Mediator** обновляет локальные записи: заменяет временные ID, меняет статус на `SYNCED`, фиксирует `last_modified_at`
11. Система показывает уведомление «Загрузка завершена» и обновляет список фото в альбоме

### Альтернативные потоки
| ID | Условие | Действия системы |
|----|---------|------------------|
| A1 | Файл превышает лимит 20 МБ или имеет неподдерживаемый формат | Отклоняет файл, показывает Toast «Формат/размер не поддерживается», предлагает сжать → Возврат к шагу 2 |
| A2 | Отсутствует подключение к интернету | Сохраняет только в Room со статусом `PENDING_SYNC` → Планирует отправку через `WorkManager` при появлении сети → Завершает сценарий с уведомлением «Сохранено локально» |
| A3 | Ошибка сервера (5xx / 413 / 401) | Помечает запись как `SYNC_FAILED` → Повторяет отправку с экспоненциальной задержкой (5м, 15м, 60м) → После 3 неудач показывает баннер «Ошибка синхронизации» |
| A4 | Пользователь отменяет загрузку нажатием «Отмена» | Откатывает транзакцию в Room, удаляет временные записи и кэш превью → Возвращает на экран альбома |

### Бизнес-правила
- Максимум 10 файлов за одну операцию загрузки
- Автоматическое извлечение EXIF-данных (камера, выдержка, ISO) при первом сохранении
- При оффлайн-режиме приоритет отдаётся локальному сохранению, синхронизация откладывается
- Удаление фотографии из альбома не удаляет файл с сервера до подтверждения в настройках приватности

### Маппинг на слои PCMEF
| Слой | Компонент | Ответственность |
|------|-----------|-----------------|
| **Presentation** | `PhotoPickerScreen`, `AlbumDetailScreen` | Отображение галереи, индикаторов загрузки, уведомлений |
| **Control** | `PhotoUploadViewModel` | Валидация ввода, управление `StateFlow`, обработка UI-событий |
| **Mediator** | `PhotoSyncUseCase`, `ExifParser` | Координация генерации превью, извлечение метаданных, управление статусами |
| **Foundation** | `PhotoDao` (Room), `PhotoApiService` (Retrofit), `WorkManager` | Локальное кэширование, сетевые запросы, фоновая отправка |

## UC6: Оффлайн-кэширование и синхронизация данных

| Параметр | Описание |
|----------|----------|
| **ID** | UC6 |
| **Название** | Оффлайн-кэширование и синхронизация данных |
| **Актор** | Система (автоматический процесс) + Фотограф (инициатор ручного обновления) |
| **Краткое описание** | Обеспечение работы приложения без подключения к интернету за счёт локального кэша (Room) и автоматической двусторонней синхронизации изменений с сервером при восстановлении соединения. |

### Предусловия
1. Приложение запущено, схема `RoomDatabase` инициализирована
2. Пользователь ранее успешно вошёл в систему, токен сохранён в `EncryptedSharedPreferences`
3. Включён `ConnectivityManager` для отслеживания состояния сети

### Постусловия
- **Успех:** Локальная и серверная БД согласованы, все изменения применены, статус записей `SYNCED`, зафиксировано `lastSyncTimestamp`
- **Неуспех:** Конфликты разрешены по стратегии LWW, пользователь уведомлён о неразрешимых расхождениях, транзакция откатана

### Триггер
- **Автоматический:** Смена состояния сети на `CONNECTED` или срабатывание `WorkManager` по расписанию (каждые 4 часа)
- **Ручной:** Пользователь выполняет `Pull-to-Refresh` на экране списка или нажимает «Синхронизировать» в `SettingsScreen`

### Основной поток
1. Система определяет направление: `PUSH` (клиент→сервер) и `PULL` (сервер→клиент)
2. **Push-фаза:** `SyncDao` запрашивает записи со статусами `PENDING_SYNC` или `SYNC_FAILED`
3. Слой **Mediator** формирует пакет изменений, применяет стратегию разрешения конфликтов (сравнение `last_modified_at`)
4. `Retrofit` отправляет пакет на `POST /api/sync` с заголовком `Authorization: Bearer <token>`
5. Сервер применяет изменения, возвращает подтверждённые версии и новые/обновлённые записи
6. **Pull-фаза:** Клиент запрашивает инкрементальные обновления через `GET /api/sync?since=<lastSyncTimestamp>`
7. Слой **Mediator** валидирует полученные данные, применяет бизнес-правила предметной области
8. Слой **Foundation** обновляет `Room` через транзакцию `RoomDatabase.runInTransaction()`, помечает записи как `SYNCED`
9. `ViewModel` передаёт обновление в UI через `StateFlow`, списки перерисовываются
10. Система сохраняет новое `lastSyncTimestamp` в `DataStore` и скрывает индикатор синхронизации

### Альтернативные потоки
| ID | Условие | Действия системы |
|----|---------|------------------|
| A1 | Обнаружен конфликт версий (клиент и сервер изменили одну запись) | Применяет стратегию LWW (приоритет сервера) → Записывает событие в `SyncLog` → Показывает SnackBar «Данные обновлены с сервера» |
| A2 | Сервер недоступен или таймаут (>15 сек) | Помечает попытку как `RETRY_SCHEDULED` → `WorkManager` планирует повтор через 5, 15, 60 мин → Завершает сценарий |
| A3 | Токен авторизации истёк (401 Unauthorized) | `Control` инициирует refresh-поток через `POST /api/auth/refresh` → При успехе повторяет синхронизацию → При неудаче требует повторного входа |
| A4 | Пользователь принудительно отменяет синхронизацию | Прерывает `CoroutineScope`, откатывает незавершённую транзакцию в Room → Возвращает UI в стабильное состояние |

### Бизнес-правила
- Синхронизация работает только для авторизованных пользователей
- Приоритет при конфликте: серверная версия (Server-Wins), кроме полей `draft` и `local_only`
- Максимальный размер пакета синхронизации: 50 записей за запрос
- История конфликтов хранится 30 дней для аудита и отладки
- Оффлайн-режим поддерживает полный CRUD для `Photo`, `Album`, `Portfolio`

### Маппинг на слои PCMEF
| Слой | Компонент | Ответственность |
|------|-----------|-----------------|
| **Presentation** | `SyncStatusIndicator`, `PullToRefresh`, `SettingsScreen` | Визуализация статуса, ручные триггеры, уведомления |
| **Control** | `SyncViewModel`, `NetworkStateObserver` | Отслеживание сети, управление состоянием UI, обработка отмены |
| **Mediator** | `SyncOrchestrator`, `ConflictResolver`, `SyncRepository` | Координация PUSH/PULL, разрешение конфликтов, бизнес-валидация |
| **Foundation** | `SyncDao` (Room), `SyncApiService` (Retrofit), `WorkManager`, `DataStore` | Локальные запросы, сетевые вызовы, фоновое планирование, хранение метаданных синхронизации |
